# MVP Implementation Prompt: Employee Transportation VRP Solver (Replit Edition)

## Project Context

Build a **Minimum Viable Product (MVP)** for an employee transportation routing system using **Quarkus 3.17.3**, **Timefold Solver 1.17.0**, and **GraphHopper** routing engine with **Java 17**. This system optimizes driver routes for transporting employees between a central hub (Hauptbahnhof Worms) and multiple customer locations with complex scheduling requirements.

## Problem Overview

### Core Business Problem
Transport employees from a central hub to various customer locations for shift work, optimizing for:
1. **Minimize total distance** (highest priority - reduces gas and vehicle wear)
2. **Maximize consecutive working hours** for drivers (soft constraint - aim for 4-hour blocks)
3. **Minimize employee travel time** (lower priority)

### Key Constraints
- **Hard Constraints:**
  - Vehicle capacity: 6 passengers per vehicle
  - Time windows: Employees must arrive within 30-45 minutes before shift start
  - Work hours: Max 10 hours/day, 40 hours/week per driver
  - Driver breaks: 30 min minimum, 4 hours maximum between routes
  - Pickup-dropoff pairing: Dropoffs must occur after corresponding pickups

- **Soft Constraints:**
  - Minimize total distance traveled (HIGH priority)
  - Maximize consecutive working hours (aim for 4-hour blocks)
  - Minimize total travel time
  - Minimize waiting time

## Routing Engine: GraphHopper

### Why GraphHopper?
- ✅ **Native Java integration** - No external HTTP service required
- ✅ **Self-contained** - Runs in same JVM as Quarkus
- ✅ **Offline routing** - Download OSM data once, works without internet
- ✅ **Replit compatible** - Entire stack runs in one container
- ✅ **Accurate** - Real-world road network distances and times
- ✅ **Maven dependency** - Simple `pom.xml` addition

### Maven Dependency
```xml
<dependency>
    <groupId>com.graphhopper</groupId>
    <artifactId>graphhopper-core</artifactId>
    <version>8.0</version>
</dependency>
```

### GraphHopper Setup
```java
// Initialize once at startup
GraphHopper hopper = new GraphHopper();
hopper.setOSMFile("germany-latest.osm.pbf"); // ~3GB download
hopper.setGraphHopperLocation("./graphhopper-cache");
hopper.setProfiles(new Profile("car").setVehicle("car").setWeighting("fastest"));
hopper.importOrLoad();

// Use for routing
GHRequest request = new GHRequest(lat1, lon1, lat2, lon2)
    .setProfile("car");
GHResponse response = hopper.route(request);
PathWrapper path = response.getBest();
long distanceMeters = Math.round(path.getDistance());
long timeMillis = path.getTime();
```

### Integration Strategy
1. **Startup:** Download OSM extract for Rhineland-Palatinate/Hesse region (smaller than full Germany)
2. **Cache:** Store routing graph in `./graphhopper-cache` directory
3. **Distance Calculator:** Replace Haversine with GraphHopper actual routing
4. **Fallback:** Use Haversine if GraphHopper fails (for robustness)

## Event Generation Logic

For each `ShiftDemand` (when active):
1. **Always generate pickup event:** Hub → Customer
   - `minStartTime` = shift start - earlyArrivalBufferMax (e.g., 45 min)
   - `maxEndTime` = shift start - earlyArrivalBufferMin (e.g., 30 min)
   - Duration & Distance = GraphHopper route calculation
   - Assign to employees based on `employee.assignments`

2. **Conditionally generate dropoff event:** Customer → Hub
   - Only if `requiresReturnTrip == true`
   - `minStartTime` = shift end
   - `maxEndTime` = shift end + 1 hour
   - Link to pickup via `pairedEvent`

3. **Pre-assign employees:** Each event is pre-assigned based on `ShiftDemand.assignments`

## Constraint Provider Requirements

Implement `VrpConstraintProvider` with:

### Hard Constraints
1. **Capacity constraint:** `driver.getTotalLoad() ≤ driver.maxCapacity`
2. **Time window constraint:** Events must arrive within early arrival buffer and complete before maxEndTime
3. **Pairing constraint:** If paired, dropoff must occur after pickup
4. **Break constraint:** Between routes, breaks must be 30min-4hours
5. **Work hour constraint:** Max 10h/day, 40h/week

### Soft Constraints (Priority Order)
1. **SOFT (HIGH):** Minimize total distance (meters) - **Weight: 1**
2. **MEDIUM:** Maximize consecutive working hours (seconds) - **Weight: 1**
3. **SOFT (LOW):** Minimize total travel time (seconds) - **Weight: 1/100**
4. **SOFT (LOW):** Minimize waiting time (minutes) - **Weight: 1/60**

## Variable Listeners Required

1. **ArrivalTimeUpdatingVariableListener**
   - Updates `Event.arrivalTime` based on `previousStandstill` departure time + travel time (from GraphHopper)
   - Triggers on changes to `previousStandstill` or `driver`

2. **RouteSplitterListener**
   - Splits `Driver.events` into multiple `Route` objects when consecutive hours exceed 4h
   - Automatically inserts break periods between routes
   - Triggers on changes to `Driver.events`

3. **HoursTrackingListener**
   - Calculates `totalDailyHours` and `totalWeeklyHours` from routes
   - Groups by date and aggregates durations
   - Triggers on changes to `Driver.routes`

4. **TotalMetricsListener**
   - Calculates distance, travel time, consecutive hours, waiting time
   - Uses GraphHopper distances from events
   - Waiting time = sum of (minStartTime - arrivalTime) for early arrivals
   - Triggers on changes to `Driver.routes`

## Route Structure Requirements

Routes can be **fully interleaved** - no strict sequential phases required:
- ✅ Valid: Home → Pickup1 → Pickup2 → Dropoff1 → Pickup3 → Dropoff2 → Dropoff3 → Home
- ✅ Valid: Home → Pickup1 → Customer1 → Pickup2 → Customer2 → Pickup3 → Home
- ❌ Invalid: Dropoff before its paired pickup
- ❌ Invalid: Exceeding 4 consecutive hours without a break

**Break insertion:** After ~4 hours of driving, create a new route with a break period (30min-4hours) in between.

## Key Design Decisions for MVP

1. **GraphHopper for routing:** Real-world distances, fully integrated, no external dependencies
2. **JPA entities:** Customers, Employees, Shifts stored in H2 database
3. **Quarkus Qute + HTMX:** Lightweight UI without npm/webpack complexity

5. **Single-week planning:** Focus on one week at a time (expandable later)
6. **Pre-assigned employees:** Employees assigned to shifts via UI, solver assigns to drivers
7. **Bootstrap data:** Demo data loaded on first startup for testing
